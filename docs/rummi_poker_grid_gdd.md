# Rummi Poker Grid — Game Design Document (Revised)

---

# 1. 게임 개요

> 숫자 + 색상 타일을 배치하여 포커 핸드를 만들고,
> 제거 타이밍을 전략적으로 선택하는 퍼즐 게임

## 1.1 v1 고정 규칙 요약

| 항목 | 확정 내용 |
|------|-----------|
| 평가 라인 | **행 5 + 열 5 + 주·반 대각 2 = 12줄** |
| 빈 칸이 있는 줄 | **판정 안 함** — 5칸이 모두 찰 때만 평가 |
| Straight | 일반 연속 `n..n+4` + **10–11–12–13–1** |
| 죽은 줄 | **하이카드·원페어** |
| 죽은 줄 처리 | **보드패 버림으로만 완화** |
| 일괄 확정 | 대기 중인 완성 줄 **전부**를 한 번에 점수·제거 |
| 조커 | **메타 슬롯만** — 보드 타일 모델에 포함하지 않음 |
| 덱·만료 | `copiesPerTile` 기반, 리셔플 없음. **현재 덱 소진** 시 만료 |

---

# 2. 핵심 시스템

## 2.1 보드
- 5 x 5 고정 그리드
- 카드 이동 없음
- 중력 없음

---

## 2.2 타일·덱 (포커 v1)

- 숫자(랭크): **1 ~ 13** (A=1 … K=13)
- 색(슈트): 🔴 🔵 🟡 ⚫ — 스페이드/하트/다이아/클럽에 대응 가능
- **덱 장수**는 `copiesPerTile` 기반: 총 **4색 × 13랭크 × copiesPerTile**. `Tile.id`로 물리 복제 타일을 구분한다.
- 즉 **52장(1복제) / 104장(2복제)** 을 같은 로직으로 처리 가능하다.
- **손패 기본 한도는 1장**으로 잡는다. 다만 현재 구현은 디버그 메뉴에서 `maxHandSize`를 `1~3장` 범위로 조절할 수 있다.
- **전부 소모 시 만료** — §8.4.

---

## 2.3 버림 시스템 (개정)

- 버림 자원은 현재 두 갈래다.
  - **보드 버림 \(D_{board}\)**: 배치된 보드 타일을 버릴 때 사용
  - **손패 버림 \(D_{hand}\)**: 손패 카드를 버리고 새 카드를 볼 때 사용
- 죽은 줄 완화는 **보드 버림 \(D_{board}\)** 으로만 한다.
- **버림 시 즉시 손패 보충 (덱에서 1장)** 을 기본으로 둔다.
- (구버전 메모) 스테이지 전체 4회 제한 등은 **블라인드별 \(D\)** 로 대체하는 것이 §8과 일치한다.

---

# 3. 턴 흐름

1. 카드 드로우 → 슬롯 추가
2. 플레이어 선택
   - 배치
   - 버림 (즉시 보충)
3. 보드 갱신
4. 핸드 분석
5. 제거 여부 선택

**배치·드로우 사이클(턴) 개수는 v1에서 무제한** — 족보 확정 횟수와 별개로, **주요 한도는 §8의 디스카드 \(D\)** 와 **덱 소모**로 준다.

# 4. 핸드 시스템

## 4.1 판정 범위

- 가로 5칸
- 세로 5칸
- **주 대각선·반 대각선 각 5칸** (빙고와 같이 **총 12줄** 평가)

---

## 4.2 핸드 종류 및 점수 (재정리)

| 핸드 | 조건 | 점수 | 비고 |
|------|------|------|------|
| High Card | 위 족보 없음 | 0 | **죽은 줄** (양수 점수 없음) |
| One Pair | 같은 숫자 2개 | 0 | **죽은 줄** (하이카드와 동일 그룹) |
| Two Pair | 2쌍 | 25 | ✔ |
| Three of a Kind | 3개 | 40 | ✔ |
| Straight | 연속 숫자 (일반 연속 또는 **10–11–12–13–1**) | 70 | ✔ |
| Flush | 같은 색 | 50 | ✔ |
| Full House | 3 + 2 | 80 | ✔ |
| Four of a Kind | 4개 | 100 | ✔ |
| Straight Flush | 연속 + 같은 색 | 150 | ✔ |

---

## 4.3 제거 규칙

> 핸드 완성 ≠ 자동 제거

### 제거 방식 (수동 · 일괄 확정)

- **한 번에 터뜨릴 완성 줄 개수 제한 없음.**
- **점수 확정**: **전용 버튼** 또는 **임의의 완성 평가줄 탭** — 동작 동일. **대기 중인 완성 줄 전부**를 한 번에 점수 합산 후, 각 줄에서 **족보 성립에 실제 기여한 카드만** 동시에 제거한다.
- 방치·대기에 따른 **추가 ±점** 은 밸런스 미확정. 구현은 **튜닝 변수 기본 0** (`game_logic` §2.1).

---

## 4.4 제거 결과

### ✔ 유효 핸드
- 점수 획득
- 족보 성립 카드 제거

세부 기준:
- `투페어`, `트리플`, `포카드`는 **매칭된 카드만 제거**
- `스트레이트`, `플러시`, `풀하우스`, `스트레이트플러시`는 **5장 전체 제거**

### ✔ 죽은 줄 (하이카드 · 원페어)
- 양의 점수 없음 (표상 0)
- **줄 확정으로 제거하지 않는다.**
- 죽은 줄을 완화하려면 **해당 5칸 중 보드 타일 하나를 버림(D)** 해서 줄 구성을 깨야 한다.
- **보유·방치에 대한 추가 패널티**는 밸런스 검토 후 결정 (§6.3과 중복 여부 포함)

# 7. 조커 시스템

## 기본
- **보드 타일이 아님** — **메타(장착 슬롯)** 에만 존재
- 최대 5개 장착
- 점수 보정 중심

## 현재 구현 범위

- 원본 데이터는 `data/common/jesters_common.json`에 유지한다.
- 현재 런/상점/밸런스 기준 카탈로그는 `data/common/jesters_common_phase5.json`을 사용한다.
- 현재는 **common Jester 중에서도**, 지금 보드형 점수 체인과 라운드 종료 경제 정산에 자연스럽게 맞는 카드만 curated 런타임 풀로 본다.
- 전투 점수에 직접 반영 가능한 조건형 Jester를 우선 지원
  - `chips_bonus`
  - `mult_bonus`
  - `xmult_bonus`
  - `scholar` 특수 처리
- 지원 조건
  - `none`
  - `suit_scored`
  - `pair`
  - `two_pair`
  - `three_of_a_kind`
  - `straight`
  - `flush`
  - `face_card`
  - `rank_scored`
  - `cards_remaining_in_deck`
  - `remaining_discards`
  - `owned_jester_count`
  - `zero_discards_remaining`
  - `empty_jester_slots`
- 후속 구현 대상
  - `stateful_growth`
  - `economy`
  - `rule_modifier`
  - `retrigger`

### 현재 common curated 풀 (룰 기준)

아래는 **지금 게임 구조에 맞아 1차로 인정하는 common Jester**다.  
공통 기준은 다음과 같다.

- `5x5 / 12줄 / 줄 확정 / 기여 카드 점수화` 구조와 충돌하지 않을 것
- 제출 순서, 손에 든 카드 정렬, 강화 카드, 타로/행성/바우처, 히든 핸드 해금에 의존하지 않을 것
- 현재 점수 체인 `기본 점수 -> chips_bonus -> mult_bonus -> xmult_bonus` 또는 라운드 종료 캐시아웃 정산에 자연스럽게 들어올 것

### 상점 노출 정책 (현재)

- 현재 상점은 **위 common curated 풀만** 오퍼 후보로 본다.
- 즉 데이터 파일에 common이 더 있어도, 아래 조건을 만족하지 않으면 상점 오퍼에 넣지 않는다.
  - 현재 점수 체인 또는 라운드 종료 정산에 즉시 들어갈 수 있음
  - 현재 손패 1장/보드 줄 평가 구조와 충돌하지 않음
  - 별도 소모품/성장/재트리거 시스템이 없어도 설명 가능함
- 이유: 지금 정산 루프는 `줄 확정 -> 기본 점수 계산 -> Jester 보정 -> 누적 점수 반영`까지만 안정적으로 지원하며, 메타·지속 상태·재트리거 규칙은 후속 설계가 필요하다.

### 후속 도입 순서 (룰만 고정)

지금은 구현보다 **도입 순서와 범위만 먼저 고정**한다.

1. `economy`
2. `stateful_growth`
3. `rule_modifier`
4. `retrigger`

### 7.1 1차 경제 수치 고정

v1에서 우선 고정하는 경제 수치는 아래와 같다.

| 항목 | 현재 고정값 |
|------|-------------|
| 시작 골드 | `10` |
| 스테이지 클리어 기본 보상 | `10 Gold` |
| 남은 `D_board` 보너스 | `+5 Gold` / 1회 |
| 남은 `D_hand` 보너스 | `+2 Gold` / 1회 |
| 상점 기본 리롤 비용 | `5 Gold` |
| 같은 상점 내 리롤 증가 | `+1 Gold`씩 누적 |
| 상점 진입 시 리롤 비용 | 다시 `5`로 초기화 |
| 상점 오퍼 수 | 기본 `3`장, 풀 부족 시 가능한 만큼 |
| 구매가 | 카드 `baseCost` 그대로 사용 |
| 판매가 | `floor(baseCost / 2)`, 최소 `1 Gold` |

의도:
- `D_board`는 죽은 줄 완화에 직접 연결되므로 `D_hand`보다 가치가 높다.
- 구매가는 데이터 파일의 카드별 개성을 유지하고, 판매가는 단순한 절반 규칙으로 고정한다.
- 리롤/오퍼 수는 먼저 단순 고정으로 두고, 할인/오퍼+1 같은 효과는 후속 `economy` 카드에서만 연다.

### 7.2 도입 순서별 실제 범위

- `economy`
  - 1차: 위 **기본 경제 수치 고정**을 코드/문서에 반영
  - 2차: **라운드 종료형 economy Jester** 만 제한 도입
  - 제외: 부채, 확률성 돈 획득, 버림 패턴 추적, 동적 타깃 랭크
- `stateful_growth`
  - 1차: **카드 내부 카운터 1개**만 갖는 성장형 Jester 지원
  - 성장값은 `int` 단일 수치로 제한
  - 스테이지 간 유지 여부를 카드 메타에 명시
  - 현재 구현 카드: `green_jester`, `supernova`, `popcorn`, `ice_cream`, `ride_the_bus`
  - 보류 카드: `square_jester`, `red_card`, `fortune_teller`, `constellation`
- `rule_modifier`
  - 1차: **점수 공식 보정** 수준만 허용
  - 손패 크기 증가, 4장 족보 허용, 전체 플레이 카드 점수화 같은 규칙 재정의는 보류
- `retrigger`
  - 점수 이벤트 로그와 재실행 범위가 완전히 고정된 뒤 마지막 단계에서만 도입

### 7.3 economy 2차 후보와 제외 범위

현재 common 중 economy 계열은 전부 한 번에 넣지 않는다. 우선순위는 아래처럼 자른다.

- 2차 우선 후보
  - `delayed_gratification`
  - `golden_jester`
  - `egg`
- 보류
  - `credit_card`: 부채/신용 한도 규칙이 추가로 필요
  - `business_card`: 확률형 보상이라 결정적 점수/경제 테스트가 먼저 필요
  - `faceless_jester`: 동시 손패 대량 버림 조건이 현재 손패 1장 구조와 맞지 않음
  - `mail_in_rebate`: 동적 타깃 랭크 추적이 필요
  - `to_do_list`: 라운드별 목표 족보 상태가 추가로 필요
  - `reserved_parking`: held card 개념이 약함
  - `trading_card`: 첫 버림/단일 버림 이벤트 추적이 더 필요

즉, economy 2차는 **라운드 종료 정산에만 붙는 고정 수입형 카드**부터 넣는다.

### 당장 넣지 않는 성장 시스템

발라트로의 아래 시스템은 **참고만 하고 현재 v1 룰에는 넣지 않는다**.

- 타로 카드
- 행성 카드
- 유령 카드
- 바우처
- 인장/에디션/강화 카드
- 히든 핸드 대량 해금

대신 후속으로는 아래 두 축만 축소판으로 검토한다.

- `Planet-lite`
  - 특정 족보의 기본 점수 또는 mult를 영구 상승
- `Voucher-lite`
  - 상점/런 전역 패시브
  - 예: 리롤 비용 -1, 오퍼 +1, `D_board` 최대치 +1

# 8. 발라트로식 런·블라인드 (룰 이식)

**레퍼런스**: [Balatro](https://www.playbalatro.com/) — **목표 점수·상점·런** 등 메타 리듬만 참고하고, **족보를 “낼 수 있는 횟수” 제한은 두지 않는다**(본작 v1).

## 8.1 발라트로 ↔ 본작 (수정된 매핑)

| 발라트로 | 본작 (Rummi Poker Grid) |
|----------|-------------------------|
| 블라인드 목표 칩 | **블라인드 목표 점수 \(T\)** — 누적 총점이 이 값 이상이면 클리어 |
| 핸드(Hands) | **본작에서는 제한 없음** — 양수 족보가 나오는 **일괄 확정**은 횟수 제한 없이 반복 가능 |
| 디스카드 | **보드 버림 \(D_{board}\)** + **손패 버림 \(D_{hand}\)** — 블라인드마다 남은 횟수(§2.4) |
| 죽은 줄 처리 | **줄 확정 제거 없음** — **보드 버림 \(D_{board}\)** 으로 5장 중 1장을 비워 죽은 줄을 완화 |
| 팔로우 | **배치용 턴** — v1에서 **개수 무제한**. 압박은 **\(D\)**·목표 \(T\)·덱 소모로 준다 |
| 상점·달러 | 블라인드 보상 → 정산 바텀시트 → **Jester 상점 전체 화면** (§7) |
| 보스 블라인드 | 변칙 조건 — 밸런스에서 별도 정의 |

## 8.2 블라인드(라운드) 단위 상태

각 블라인드 시작 시 **리셋**되는 것(수치는 밸런스).

- **목표 점수** \(T\)
- **남은 보드 버림** \(D_{board}\) — 0이면 보드 타일 버리기로 죽은 줄 완화 불가
- **남은 손패 버림** \(D_{hand}\) — 0이면 손패 리롤 불가

**양수 족보(점수 나오는 줄)만** 모인 일괄 확정에는 **별도 횟수 제한 없음**.

## 8.3 점수·제거 누적

- **양수 족보**가 포함된 일괄 확정: 줄마다 기본 점수 + 적용 가능한 Jester 보정 점수 → 블라인드 누적에 가산. **확정 횟수 제한 없음.**
- **죽은 줄**은 일괄 확정 대상이 아니다. 점수도 없고 줄 제거도 없다. **완화 방법은 보드 버림 \(D_{board}\)** 뿐이다.

## 8.4 클리어·만료 (v1 임시)

- **클리어**: 누적 점수 ≥ \(T\) 이더라도, 먼저 **실시간 줄 정산을 모두 마친 뒤** `Stage Clear` 연출과 정산을 진행하고, 이후 상점으로 이동.
- **만료(강제 진행 종료 트리거)** — 아래 **어느 하나라도** 먼저 도달하면, 그 시점에서 블라인드·판정은 **후속 로직으로 연결**(승패·재시도 등은 차후 확정).
  1. **\(D_{board}\) = 0** 인 상태에서 **보드 25칸이 모두 타일로 찰 때**까지 이어진 경우(배치용 턴은 무제한). 이때 죽은 줄도 버림으로만 완화 가능하므로 더 이상 판을 풀 수 없다.
  2. **현재 사용 중인 덱이 모두 소모**된 경우 — §2.2 덱이 손패·보드·버림 더미 등에 모두 나간 뒤 **더 이상 뽑을 카드가 없을 때**(리셔플 없음 전제). 총 장수는 `copiesPerTile`로 결정된다. **보드 조커 타일 없음**(§7 메타만).

### 8.4.0 정산 흐름 역할 경계 (현재 코드 기준)

- `RummiPokerGridSession.confirmAllFullLines(...)` 의 책임
  - 현재 대기 중인 양수 족보 줄 전부를 일괄 점수화
  - 줄별 **기본 점수 + Jester 보정 점수** 계산
  - 족보 성립 카드만 제거
  - 블라인드 누적 점수 갱신
  - 목표 점수 달성 여부 반환
- `GameView` 의 후속 책임
  - 줄별 실시간 정산 연출
  - 활성 Jester 강조/버스트
  - `Stage Clear` 오버레이
  - `Cash Out` 바텀시트
  - `Jester Shop` 전체 화면 라우트
  - 다음 스테이지 진입 시 `runSeed + stageIndex` 기반 파생 시드로 덱 전체 리셋/셔플
- 즉 **실시간 줄 정산**과 **최종 캐시아웃/상점/다음 스테이지 준비**는 같은 흐름 안에 있지만, 책임 레이어는 분리한다.

### 8.4.1 덱 소모 속도 (설계 유의)

보드는 **최대 25칸**이지만, **줄 확정으로 칸이 비면** 그 타일은 보통 **스코어/제거 더미**로 가고 **다시 드로우 풀에 들어가지 않는다**(리셔플 없음 전제). 그러면 빈 칸에 **새 카드를 깔기 위해** 드로우가 반복되므로, **확정·클리어가 잦을수록** “덱에서 꺼낸 장수” 누적이 빨라져 **덱 소진 만료가 이르게** 올 수 있다.

- **평가만 자주 되고 확정을 아끼면** 보드에 카드가 오래 머물러 **같은 25장**만 돌리는 구간이 길어질 수 있음(드로우 소모는 상대적으로 느림).
- **확정을 자주 하면** 보드 회전이 빨라져 **추가 드로우**가 많아지고, **덱 소진 만료**가 빨리 올 수 있음.

따라서 **목표 점수 \(T\)**, **\(D\)**, **손패 기본 1장·드로우 규칙**은 플레이 테스트로 맞추고, 필요 시 **① 블라인드마다 52/104 중 어떤 덱 크기를 쓰는지 규칙화**, **② 만료 조건에서 덱 소모의 비중 조정**, **③ 손패 기본값과 디버그 허용 범위를 분리 명시**, **④ 확정으로 나간 카드의 처리(예: 나중에 리셔플 허용 여부)** 등을 **후속 룰**로 정한다. v1 문서는 **이 긴장**을 인지한 상태에서 수치를 잡는다.

## 8.5 런 구조

- 앤티 단위로 소/빅/보스 등 순환, 상점·보스 변칙은 기존과 동일하게 확장 가능.
- **발라트로와의 차이**: **핸드 제한 없음**; 긴장은 **디스카드·목표 점수·보드 만재·현재 덱 소진**에 둔다.
