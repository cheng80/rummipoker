# Rummi Poker Grid — 게임 로직 (구현 스펙)

> 본 문서는 `docs/rummi_poker_grid_gdd.md`를 **코드 작성 기준**으로 풀어 쓴 것이다.  
> GDD와 충돌 시 **GDD를 먼저 수정**한 뒤 본 문서를 맞춘다.

---

## 1. 엔티티

### 1.1 타일 (Tile)

- **숫자** `n`: 정수, 범위 **1 ~ 13** (포커 랭크)
- **색** `c`: 네 슈트 (구현: `enum TileColor { red, blue, yellow, black }`)
- 덱 장수는 **`copiesPerTile` 기반**: 총 `4색 × 13랭크 × copiesPerTile`.
- 즉 `(n, c)` 조합당 `copiesPerTile`장이고, `Tile.id`는 동일 숫자/색의 물리 복제 타일 구분용이다.
- 따라서 **52장(`copiesPerTile = 1`) / 104장(`copiesPerTile = 2`)** 을 같은 알고리즘으로 지원한다.

### 1.2 보드 (Board)

- **5 × 5** 고정 그리드
- **칸 상태**: `Empty | Occupied(Tile)`
- **이동 없음**, **중력 없음**

### 1.3 웨이스트 (임시 슬롯, Waste)

- **최대 1칸**

### 1.4 덱 (Deck)

- v1 기본값은 현재 `copiesPerTile = 1` 이다. 즉 현재 기본 세팅은 **52장**이며, **드로우 더미가 비면** §5.2 **만료(덱 소진)** — 리셔플 없음.
- 버림 후 즉시 1장 보충 등은 엔진 이벤트로 모델링.

### 1.5 시드·셔플 규칙

현재 구현은 **시드 기반 결정적 셔플**을 사용한다. 같은 코드 버전, 같은 `runSeed`, 같은 `copiesPerTile`이면 **초기 덱 순서가 동일**해야 한다.

**표준 덱 원본 순서**
- `TileColor.values` 순서대로 색을 순회
- 각 색마다 숫자 `1..13`
- 각 `(색, 숫자)` 조합에 대해 `copiesPerTile`만큼 복제
- 이 원본 리스트를 `List.shuffle(Random(seed))`로 섞는다

**런 시작 시**
- `/game?seed=...` 의 `seed`를 우선 사용
- 없으면 `RummiPokerGridSession.rollNewRunSeed()`로 새 런 시드 생성
- 세션 생성 시 `Random(runSeed)`를 만들고, `PokerDeck.shuffled(Random(runSeed), null, deckCopiesPerTile)`로 초기 덱 셔플

**드로우 방향**
- 내부 덱은 셔플된 리스트로 저장
- 실제 드로우는 리스트의 **끝**에서 `removeLast()`로 1장씩 꺼낸다
- 즉 “덱 맨 위”는 내부적으로는 **셔플된 배열의 마지막 원소**다

**다음 스테이지 진입 시**
- 이전 스테이지 덱을 이어서 쓰지 않고, **표준 덱 전체를 다시 만들고 다시 섞는다**
- 셔플 시드는 `deriveStageShuffleSeed(runSeed, stageIndex)`를 사용한다
- 현재 공식:

```dart
final mixed =
    (runSeed * 1103515245 + 12345 + stageIndex * 1013904223) & 0x7fffffff;
return mixed == 0 ? stageIndex + 1 : mixed;
```

- `stageIndex`는 **다음 스테이지로 증가한 값**을 사용한다
- 따라서 같은 `runSeed`로 시작한 런은 각 스테이지 덱 순서도 재현 가능해야 한다

**현재 의도**
- 초기 스테이지 재현: `runSeed`
- 스테이지별 덱 재현: `runSeed`와 `stageIndex`를 `deriveStageShuffleSeed(...)` 공식으로 섞어 만든 **파생 시드** 기반 결정적 셔플
- 덱 장수 변경 대응: `copiesPerTile`만 바꾸고 같은 셔플 규칙 유지

향후 셔플 공식을 바꾸면, 이 절과 `RummiPokerGridSession.deriveStageShuffleSeed`, `PokerDeck.shuffled`, 관련 테스트를 함께 갱신한다.

## 2. 턴 상태 머신 (개요)

GDD §3와 동일: 드로우 → 배치/버림 → 보드 갱신 → 핸드 분석 → (UI) 제거 선택.

### 2.1 평가 유지·일괄 점수화 (권장)

**문제**  
한 칸은 **행·열·대각**에 동시에 속할 수 있다. **완성 줄을 하나씩만 제거**하면, 첫 제거에서 공유 타일이 사라져 **나머지 줄의 5장이 깨져** 다음 줄 족보가 무효가 된다.  
또 **줄마다 따로 눌러 합산**하면, 플레이어 조작이 늘고 **제거 순서**에 따라 결과가 달라져 공정성·UX가 나빠진다.

**점수 합산**  
각 완성 줄은 **서로 다른 5장 집합**에 대한 족보이므로, 줄마다 점수를 매긴 뒤 **단순 합산**하면 된다(같은 타일을 ‘점수 두 번’ 치는 구조는 아님).

**확정 규칙**  
- **한 번에 터뜨릴 줄 개수 제한 없음.**  
- **확정 트리거**: **전용 버튼**이든 **임의의 완성 평가줄 탭**이든 동일 — **현재 대기 중인 완성 줄 전부**를 빙고처럼 **한 번에** (1) 줄마다 족보 점수 합산 (2) 해당 줄에서 **족보 성립에 실제 기여한 카드 칸의 합집합만** 동시 제거.  
- 보드는 완성 줄을 **대기**로 표시할 수 있고, 사용자는 크로스·대각을 맞출 때까지 유지 가능.

**턴 위험·가산 (밸런스 미확정 — 코드 플레이스홀더)**  
- 죽은 줄 방치·평가줄 대기 등 **턴 경과에 따른 ±** 는 플레이 테스트 후 결정.  
- 구현: `LineHazardTuning` — 기본 **가산 0** (효과 없음). 추후 **+10 / -20** 등 수치만 바꿔 적용.  
- 나중에 **곱**이나 복합 규칙이 필요하면 **숫자 필드 대신** `double Function(…)` 형태의 **수식/전략 객체**로 교체하는 것을 전제로 한다.

## 3. 평가 라인 (Hand scope)

**확정**

- **행 5 + 열 5 + 주 대각선 + 반 대각선 = 총 12줄**
- 각 줄은 **정확히 5장**이 모두 있을 때만 족보 판정
- 빈 칸이 있으면 그 줄은 **평가하지 않음**

구현: `RummiBoard.row` / `col` / `diagMain` / `diagAnti` + `RummiPokerGridEngine.evaluate*`.

## 4. 포커 핸드 판정

한 줄 5장에 대해 **최상위 족보 1개**.

| 족보 | 점수(양수) 비고 |
|------|-----------------|
| High Card | 0 — **죽은 줄** |
| One Pair | 0 — **죽은 줄** |
| Two Pair | 25 |
| Three of a Kind | 40 |
| Straight | 70 — **일반 연속** 또는 **10–11–12–13–1** 휠 |
| Flush | 50 |
| Full House | 80 |
| Four of a Kind | 100 |
| Straight Flush | 150 |

### 4.1 Straight

- **일반**: 서로 다른 숫자 5개가 `n, n+1, …, n+4` (1≤n≤9)
- **휠**: `1, 10, 11, 12, 13` (10–J–Q–K–A, A=1)
- **Straight Flush**: 위 스트레이트 + 5장 같은 색

### 4.2 죽은 줄

- **하이카드**와 **원페어**는 양의 점수가 없음 → `isDeadLineRank`
- 보유·방치에 대한 **추가 패널티**는 밸런스 단계에서 결정

### 4.3 조커

- **보드 타일 아님** — **메타 장착 슬롯**만 (GDD §7). 타일 모델에 조커 필드 두지 않음.
- 카탈로그는 `data/common/jesters_common.json`에서 읽는다.
- 현재 구현은 전투 점수에 직접 작용하는 조건형 Jester만 우선 지원한다.

## 5. 제거·진행

### 5.1 제거 (Clear line)

- 자동 제거 없음. **§2.1**의 일괄 확정 시, 선택된 완성 줄들에 대한 칸 제거를 **한 트랜잭션**으로 처리한다.
- 제거 범위는 **줄 전체가 아니라 족보 성립 카드만**이다.
  - `투페어` / `트리플` / `포카드`: 매칭 카드만 제거
  - `스트레이트` / `플러시` / `풀하우스` / `스트레이트플러시`: 5장 전부 제거
- 5칸 완성 줄만 “완성·대기·확정 제거”의 대상 (판정과 동일 조건).

### 5.2 블라인드·런 (GDD §8)

**상태 (블라인드마다 리셋하는 예시 필드)**

| 필드 | 의미 |
|------|------|
| `targetScore` | 목표 점수 \(T\) |
| `discardsRemaining` | 남은 **버림** \(D\) |
| `scoreTowardBlind` | 이번 블라인드 누적 점수 |
| `drawPileRemaining` (또는 동등) | **뽑을 수 있는 카드** — 초기값은 `copiesPerTile`에 따라 달라지며, 0이면 **덱 소진 만료** |

**규칙**

- **클리어**: `scoreTowardBlind >= targetScore` 이더라도, 먼저 **현재 실시간 줄 정산 체인**을 끝까지 처리한 뒤 스테이지 클리어 연출/정산/상점 이동을 진행한다.
- **양수 족보 일괄 확정**: 횟수 제한 없음.
- **죽은 줄 처리**: 하이카드·원페어는 **일괄 확정으로 제거하지 않는다**. 죽은 줄을 완화하려면 **해당 5칸 중 보드 타일 하나를 버림(D)** 해서 라인을 깨야 한다.
- **만료(v1)**: (1) `D == 0` 인 상태에서 보드 **25칸 전부 점유** 도달, 또는 (2) **현재 덱 모두 소모**로 `drawPileRemaining == 0` (`copiesPerTile`에 따라 총 장수 결정, 손 1장·보드·웨이스트·버림 더미 등 합산은 덱 모듈에서 정의). 판정·승패는 후속.
- **턴 수(배치 사이클)**: 무제한(자원은 \(D\)·덱).
- **페이싱**: 확정으로 보드에서 제거된 카드가 **드로우 풀로 복귀하지 않으면**, 클리어가 잦을수록 **추가 드로우**가 늘어 **덱 소모 만료가 빨리** 올 수 있다 — GDD §8.4.1. \(T\)·\(D\)·손패 규칙은 밸런스로 조정.
- **Jester 점수화**: `confirmAllFullLines(jesters: ...)` 가 각 줄의 기본 점수에 Jester 보정값을 더해 최종 점수를 산출한다. 현재 지원 범위는 `chips_bonus`, `mult_bonus`, `xmult_bonus`, `scholar` 및 그에 연결된 조건형 판정들이다.

**현재 코드**: `RummiPokerGridSession`이 `PokerDeck`·`RummiBlindState`·일괄 확정·만료 신호를 제공하고, `GameView`가 Flutter 위젯으로 HUD/보드/손패 입력, 실시간 정산, 상점 흐름을 담당한다. Flame은 필요 시 연출 레이어로만 재도입한다.

**UI**: 타이틀에서 **랜덤 시드** 또는 **시드 입력**으로 `/game?seed=…` 진입. 메인 게임 화면은 상단 제목줄 없이 플레이 공간을 우선하고, **시드 확인/복사 UI는 옵션 다이얼로그 내부**에 둔다. 스테이지 진행 중 상점은 **전체 화면 라우트**로 열리고, 쿼리에 `seed` 없으면 `RummiPokerGridSession.rollNewRunSeed()`로 채움.
