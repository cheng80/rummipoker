# Rummi Poker Grid — 다음 작업 준비 메모

> `docs/` 문서 갱신본을 기준으로, **현재 코드 상태와의 차이**를 정리하고 **바로 다음 구현 순서**를 고정한다.

---

## 1. 문서 기준으로 확정된 큰 변화

- 게임 중심 규칙은 이제 **5x5 + 12줄 평가 + `copiesPerTile` 기반 덱**으로 고정.
- 진행 압박은 **블라인드 목표 점수 `T`**, **버림 `D`**, **덱 소모**로 준다.
- **족보 확정 횟수 제한은 없음**. 한 번의 확정에서 **대기 중인 완성 줄 전부**를 일괄 점수화/제거한다.
- 만료 조건은 v1 기준
  - `D == 0` 인 상태에서 보드 25칸 만재
  - 또는 **현재 드로우 덱 소진**(`copiesPerTile` 기준 총 장수)
- 조커는 **보드 타일이 아니라 메타 슬롯**이다.
- 클리어 후에는 **실시간 줄 정산 -> Stage Clear 판정 -> Cash Out Bottom Sheet -> Jester 상점 전체 화면 -> 다음 스테이지** 흐름이 현재 Flutter UI에 연결되어 있다.
- 다음 스테이지 진입 시 덱은 **`copiesPerTile` 기준 전체 리셋**되고, **런 시드 기반 파생 셔플**로 재현 가능하게 맞춰져 있다.
- Jester 카드명/효과는 현재 **한글 우선 리소스 번역 스코프**를 통해 UI에 반영되고 있다.

---

## 2. 현재 코드와 문서의 핵심 차이

### 2.1 정합성 결론

- 사용자 확정 규칙은 **죽은 줄은 줄 확정으로 지우지 않고, 보드 버림(`D`)으로만 완화**하는 것이다.
- 현재 `RummiBlindState` / `RummiPokerGridSession` / 테스트 방향은 이 규칙과 대체로 맞다.
- 문서 쪽에 남아 있던 **이전 죽은 줄 제거 자원 규칙 흔적**을 걷어내는 것이 우선이다.

### 2.2 UI/플레이 루프 미완성 영역

- 웨이스트 슬롯 UI가 없음
- Riverpod 세션 연결이 아직 없음
- 타이틀/앱 설정/번역/스토어 메타데이터 정리가 아직 끝나지 않음
- Jester의 `stateful_growth / economy / rule_modifier / retrigger` 계열은 아직 전투/메타에 완전 적용되지 않음
- 정산 체인은 붙었지만, 실시간 줄 정산과 최종 정산의 책임 분리가 더 명확해질 여지가 있음

## 3. 바로 착수할 작업 순서

### 작업 1. 정산 체인과 Jester 효과 범위를 문서/코드에서 명확히 맞춘다

대상:
- `lib/logic/rummi_poker_grid/jester_meta.dart`
- `lib/logic/rummi_poker_grid/rummi_poker_grid_session.dart`
- `lib/views/game_view.dart`
- `test/logic/rummi_session_test.dart`
- 필요 시 `docs/rummi_poker_grid_game_logic.md`

해야 할 일:
- 실시간 줄 정산, Jester 강조/버스트, 스테이지 클리어 판정, 캐시아웃의 순서를 문서와 코드에서 같은 말로 정리한다
- 현재 구현된 조건형 Jester와 미구현 Jester를 문서/코드에서 같은 범위로 말하게 맞춘다
- 점수 환산식이 현재 블라인드 스케일에 맞는지 검증한다
- 상점에 노출되지만 효과 미구현인 카드가 있다면 정책을 정한다

완료 기준:
- 세션/테스트/문서 설명이 모두 **현재 적용되는 Jester 범위와 정산 순서**를 동일하게 말한다.

### 작업 2. 메타 진행과 경제 규칙을 정교화한다

대상:
- `lib/logic/rummi_poker_grid/jester_meta.dart`
- `lib/views/game_view.dart`
- 필요 시 신규 메타 상태 파일

해야 할 일:
- 상점 리롤/구매/판매 정책을 문서 기준으로 고정
- 클리어 보상, 정산 수치, 다음 스테이지 진입 흐름을 정리
- 상점이 **전체 화면**이라는 전제에서 내 Jester 슬롯 확인/상세/판매 동선을 더 다듬는다
- 필요하면 상점 노출 후보를 실제 지원 가능한 Jester 중심으로 재조정

완료 기준:
- 메타 진행 수치와 UI 흐름이 문서와 코드에서 같은 말을 한다.

### 작업 3. Flutter 셸과 상태 연결을 정리한다

대상:
- `lib/views/game_view.dart`
- `lib/providers/...` 신규 또는 기존 확장
- `docs/riverpod_architecture.md`에 맞는 연결부

해야 할 일:
- 세션 스냅샷을 Riverpod `Notifier`로 노출
- Flame은 직접 `ref`를 들지 않고 콜백/주입으로 연결
- 게임 종료/블라인드 클리어/오버레이 갱신을 Flutter와 분리

완료 기준:
- `GameView`가 세션 수명주기와 UI 오버레이를 소유하고, Flame은 렌더/입력에 집중한다.

### 작업 4. 출시 표면 정리를 한다

대상:
- `lib/views/title_view.dart`
- `lib/app_config.dart`
- `assets/translations/*.json`
- 플랫폼 표시 이름/설명 관련 파일

해야 할 일:
- 앱 카피와 진입 문구를 `Rummi Poker Grid` 기준으로 통일
- 번역 문자열 누락 정리
- 스토어 메타데이터 문서 경로 재정리

---

## 4. 추천 구현 순서

1. **Jester 효과 범위 / 메타 규칙 정합성 수정**
2. **상점 / 정산 / 경제 수치 정리**
3. **Riverpod 연결**
4. **타이틀/번역/출시 문구 정리**
5. **웹 릴리즈 검증**

이 순서를 권장하는 이유:
- 1번이 틀리면 이후 메타 루프와 점수 밸런스가 잘못된 규칙을 기준으로 고정된다.
- 2번까지 끝나야 문서상 핵심 플레이 루프를 실기기로 검증할 수 있다.
- 3번은 구조 정리지만, 규칙과 HUD가 먼저 고정되어야 상태 표면이 안정된다.

---

## 5. 착수 전 체크

- `execution_checklist`의 남은 항목이 최신 규칙(`T/D` + 죽은 줄 버림 처리)과 충돌하지 않는지 다시 본다.
- 죽은 줄은 **5칸 중 하나를 버려 라인을 깨는 방식**으로만 처리한다.
- 덱 장수가 고정 문구가 아니라 `copiesPerTile` 변수에서 결정된다는 점을 문서와 코드에서 같은 말로 유지한다.
- 상점은 더 이상 바텀시트가 아니라 **전체 화면 라우트**라는 점을 문서에서 고정한다.
